# `proposals/v1` — the only write path for agents

**Status:** published · **Version:** 1 · [CHANGELOG](CHANGELOG.md) ·
[JSON Schema](v1.schema.json)

All agent and librarian writes to the vault go through proposals. The
mechanism is **local-first and forge-free**: the validator works with no
git remote, no forge, no CI — a laptop with nothing but the `kp` binary
gets the full safety model. Forge-side validation is an optional hardening
tier that runs the exact same validator module.

## Layout

`<vault>/.kp/proposals/<ULID>/` containing:

- **`proposal.json`** —

  ```json
  {
    "schema": "proposals/v1",
    "id": "01J1PZ2M3N4P5Q6R7S8T9V0W1X",
    "created": "2026-07-03T09:15:00Z",
    "author": "kp-librarian",
    "title": "Daily digest 2026-07-03",
    "rationale": "12 new notes since last digest matched the now.md anchor.",
    "status": "open",
    "files": ["digests/2026-07-03.md"]
  }
  ```

  `id` is a ULID (also the directory name); `author` is `kp-librarian` or
  an agent-supplied name; `status` is `open | applied | rejected`; `files`
  lists the vault-relative paths touched by the patch.

- **`changes.patch`** — a unified diff against the vault working tree.

## CLI

| command | does |
|---|---|
| `kp propose` | create a proposal from staged changes or generated content |
| `kp review <id>` | human-readable render of the changeset |
| `kp apply <id>` | validate → apply → stamp status |

## Validation (hard-rejects)

The deterministic validator refuses, with no override flag:

1. any path under `.curio/**` (producer-owned state);
2. any hunk editing Curio machine frontmatter keys or content inside a
   `curio:managed` region;
3. any path outside the vault (absolute paths, `..` traversal);
4. patches that do not apply cleanly;
5. new notes whose identity is absent or duplicates an existing identity.

## Digests are proposals too

Librarian digest notes ride this same path. A digest proposal is
**auto-applicable** only when it purely ADDS files under
`[librarian].digest_dir` with `kp_id: kp:<uuidv7>`. Digests are
create-only and idempotent by date — a re-run never edits an
already-applied digest.
