# `kp-note/v1` — note identity + enrichment frontmatter

**Status:** published · **Version:** 1 · [CHANGELOG](CHANGELOG.md) ·
[JSON Schema](v1.schema.json)

The Knowledge Plane ingests any markdown note in the configured vault.
Notes MAY carry KP frontmatter; producer-generated notes (Curio, the
Zotero translator) always do:

```yaml
---
kp_id: "curio:0197b2c4-8f3e-7cc1-a5d2-3e9f10aa4b6d"   # producer-namespaced identity
kp_schema: kp-note/v1
checksum: "sha256:9f86d08..."     # change token ONLY, never identity
title: "Article title"
created: 2026-07-03T09:15:00Z
updated: 2026-07-03T10:00:00Z
tags: [rust, databases]
source: "https://example.com/article"   # null for born-in-vault notes
---
```

## Identity namespaces

`kp_id` is minted, never derived from content or location:

| namespace | form | minted by |
|---|---|---|
| `curio:` | `curio:<uuidv7>` | the Curio reader, at save time |
| `zotero:` | `zotero:<itemKey>` | Zotero (item key) |
| `kp:` | `kp:<uuidv7>` | the plane itself — born-in-plane notes, e.g. digests |
| `path:` | `path:<vault-relative-path>` | implicit fallback for plain vault notes without `kp_id` — a pragmatic v1 choice, documented as **rename-fragile** |

## Binding rules

1. **`checksum` is a change token ONLY, never identity.** Two notes with
   identical bodies are still two notes. Consumers MUST NOT key anything
   on checksum.
2. **No `status` field.** Lifecycle lives index-side, never in note
   frontmatter — producer re-exports re-render whole files and would
   silently clobber injected fields.
3. **Unknown frontmatter keys are preserved.** The block above is what the
   plane reads and writes; everything else in a note's frontmatter belongs
   to the user or other tools.

## Curio notes: adapter, not template

Curio notes are ingested via the Curio adapter: the adapter reads vanilla
`curio.frontmatter.v1` notes natively and maps `curio_id` →
`kp_id: curio:<id>`. Curio is NOT required to emit kp-note frontmatter.
Vendored, sha-pinned Curio schemas ([../vendor/curio/](../vendor/curio/))
validate at the boundary.

- **`.curio/**` is Curio-owned.** The plane never writes there; proposals
  touching `.curio/**` are hard-rejected.
- `.curio/manifest.json` is read as the write-ownership oracle:
  manifest-listed paths are Curio-owned at the managed-region level.
- KP enrichment of a Curio note = companion content BELOW the managed
  region, plus additional frontmatter keys outside Curio's machine-key
  set — never inside either.
